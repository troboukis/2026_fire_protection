# Μεθοδολογία Συλλογής και Επεξεργασίας Δεδομένων

## Σκοπός και εύρος

Το project καταγράφει και αναλύει δημόσιες δαπάνες που σχετίζονται με την πυροπροστασία στην Ελλάδα. Στόχος είναι να αποτυπωθεί πόσα χρήματα δαπανούν δήμοι, περιφέρειες και άλλοι δημόσιοι φορείς για έργα, υπηρεσίες και εξοπλισμό που συνδέονται με την πρόληψη ή την αντιμετώπιση πυρκαγιών, καθώς και πώς αυτές οι δαπάνες κατανέμονται γεωγραφικά.

Το dataset που χρησιμοποιείται στην ανάλυση περιλαμβάνει εγγραφές από το 2024 και μετά. Υπάρχουν διαθέσιμα παλαιότερα δεδομένα (από το 2009), αλλά δεν χρησιμοποιούνται στην τρέχουσα έκδοση, ώστε η σύγκριση να γίνεται σε ένα πιο συνεκτικό και πρόσφατο χρονικό διάστημα, με δεδομένα που έχουν παρόμοια μορφή και ποιότητα.

Οι βασικές πηγές δεδομένων είναι:

- το ΚΗΜΔΗΣ, για συμβάσεις προμηθειών και υπηρεσιών,
- η Διαύγεια, για αποφάσεις που σχετίζονται με δαπάνες,
- και το Copernicus EFFIS, για ιστορικά στοιχεία δασικών πυρκαγιών.

Τα δεδομένα από κάθε πηγή συλλέγονται ξεχωριστά και επεξεργάζονται σε διαδοχικά στάδια, ώστε να είναι δυνατός ο έλεγχος των ενδιάμεσων αποτελεσμάτων και να διατηρείται το αρχικό υλικό χωρίς αλλαγές.

## Πηγή 1: ΚΗΜΔΗΣ

### Συλλογή δεδομένων

Τα δεδομένα από το ΚΗΜΔΗΣ αντλούνται μέσω του OpenData API  
`cerpp.eprocurement.gov.gr/khmdhs-opendata/contract`, με χρήση του script  
`src/fetch_kimdis_procurements.py`.

Η αναζήτηση βασίζεται σε CPV codes (Common Procurement Vocabulary), δηλαδή στους ευρωπαϊκούς κωδικούς κατηγοριοποίησης προμηθειών. Χρησιμοποιούνται δύο διαφορετικές ομάδες κωδικών.

Η πρώτη ομάδα περιλαμβάνει κωδικούς που σχετίζονται άμεσα με πυροπροστασία, όπως:

- 75251100-1 — Υπηρεσίες καταπολέμησης πυρκαγιών  
- 35111000-5 — Πυροσβεστικός εξοπλισμός  
- 77312000-0 — Υπηρεσίες εκκαθάρισης από αγριόχορτα  
- 77340000-5 — Κλάδεμα δέντρων και θάμνων  

Για αυτούς τους κωδικούς κρατούνται όλες οι συμβάσεις που επιστρέφει το API.

Η δεύτερη ομάδα περιλαμβάνει πιο γενικούς κωδικούς, οι οποίοι μπορεί να περιέχουν σχετικές συμβάσεις αλλά όχι αποκλειστικά, για παράδειγμα:

- 45233141-9 — Συντήρηση οδών  
- 90910000-9 — Υπηρεσίες καθαρισμού  
- 77310000-6 — Συντήρηση χώρων πρασίνου  

Σε αυτές τις περιπτώσεις εφαρμόζεται επιπλέον φίλτρο στον τίτλο της σύμβασης.  
Η εγγραφή κρατιέται μόνο αν ο τίτλος περιέχει λέξεις-κλειδιά που σχετίζονται με πυροπροστασία, όπως  
`πυροπροστασ`, `αντιπυρικ`, `δασικών πυρκαγιών`, κ.λπ.

Ο διαχωρισμός αυτός έγινε επειδή ορισμένοι CPV είναι πολύ γενικοί και χωρίς επιπλέον έλεγχο θα περιλάμβαναν μεγάλο αριθμό άσχετων συμβάσεων.

### Incremental συλλογή

Η συλλογή γίνεται σταδιακά.

Το script κρατά την τελευταία ημερομηνία που έχει ήδη ανακτηθεί στο αρχείο  
`state/kimdis_state.json`.

Σε κάθε νέο run ζητά μόνο εγγραφές μετά από αυτή την ημερομηνία.  
Αν το state file δεν υπάρχει, η ημερομηνία εκκίνησης υπολογίζεται από τη μεγαλύτερη `submissionDate` που υπάρχει ήδη στο CSV.

Το API του ΚΗΜΔΗΣ έχει περιορισμούς στο εύρος ημερομηνιών, οπότε τα requests γίνονται σε χρονικά παράθυρα (συνήθως 180 ημερών).

Για πλήρη επαναφόρτωση χρησιμοποιείται το flag `--full-refresh`.

Σε κάθε εκτέλεση αποθηκεύεται επίσης backup των raw αντικειμένων σε JSON (`data/raw_items_backup.json`), ώστε να μπορεί να ξαναχτιστεί το CSV χωρίς νέο API call.

### Εντοπισμός και γεωκωδικοποίηση έργων

Για ένα υποσύνολο συμβάσεων του ΚΗΜΔΗΣ επιχειρείται και εντοπισμός του ακριβούς σημείου όπου αναφέρονται οι εργασίες πυροπροστασίας. Η διαδικασία υλοποιείται με το script `src/run_locate_work_updates.py` και την κλάση `Document` στο `src/locate_work.py`.

Στο στάδιο αυτό επιλέγονται μόνο συμβάσεις:

- που έχουν `reference_number`,
- που δεν έχουν ήδη εγγραφή στον πίνακα `works`,
- που ανήκουν στο τρέχον έτος,
- και που προέρχονται κυρίως από δήμους, περιφέρειες ή τον ΑΔΜΗΕ.

Για κάθε τέτοια σύμβαση γίνεται λήψη του συνημμένου PDF από το ΚΗΜΔΗΣ. Το κείμενο εξάγεται αρχικά με native PDF parsing και, όταν μια σελίδα δεν περιέχει αναγνώσιμο κείμενο, χρησιμοποιείται OCR ως fallback.

Στη συνέχεια το έγγραφο σπάει σε page-aware chunks και αποστέλλεται σε LLM prompt που ζητά να εντοπίσει μόνο σαφείς αναφορές σε συγκεκριμένα σημεία και συγκεκριμένες εργασίες πυροπροστασίας, όπως:

- αποψιλώσεις,
- κλαδέματα,
- καθαρισμούς οικοπέδων ή περιβάλλοντος χώρου,
- απομάκρυνση καύσιμης ύλης,
- διάνοιξη ή συντήρηση αντιπυρικών ζωνών.

Για κάθε εύρημα παράγονται τα πεδία:

- `point_name_raw` — η φράση όπως εμφανίζεται στο κείμενο,
- `point_name_canonical` — καθαρισμένη και γεωκωδικοποιήσιμη μορφή,
- `work` — σύντομη κανονικοποιημένη περιγραφή της εργασίας,
- `page` / `pages`,
- `excerpt` — απόσπασμα τεκμηρίωσης.

Πριν από το geocoding γίνεται deduplication ώστε να συγχωνεύονται επαναλαμβανόμενες αναφορές του ίδιου σημείου ή της ίδιας εργασίας μέσα στο ίδιο έγγραφο. Μετά, το `point_name_canonical` στέλνεται στο Google Geocoding API με περιορισμό περιοχής στην Ελλάδα. Αν βρεθεί αποτέλεσμα, αποθηκεύονται:

- `lat`, `lon`,
- `formatted_address`,
- `place_id`.

Οι τελικές εγγραφές αποθηκεύονται στον πίνακα `public.works`, ο οποίος συνδέεται με τη σύμβαση μέσω `reference_number`. Για χρήση στο frontend υπάρχει και το view `public.works_enriched`, που ενώνει τα σημεία έργων με βασικά στοιχεία της αντίστοιχης σύμβασης.

## Πηγή 2: Διαύγεια

Από τη Διαύγεια συλλέγονται αποφάσεις μέσω του search API  
`diavgeia.gov.gr/luminapi/api/search`, με χρήση λέξεων-κλειδιών που σχετίζονται με πυροπροστασία.

Τα αποτελέσματα μπορεί να περιλαμβάνουν:

- εγκρίσεις δαπάνης  
- αναλήψεις υποχρέωσης  
- απευθείας αναθέσεις  
- πληρωμές  
- αποφάσεις κατακύρωσης

Για κάθε αποτέλεσμα γίνεται δεύτερο request στο endpoint  
`/decisions/view/{ada}`  
ώστε να ανακτηθούν δομημένα μεταδεδομένα, όπως:

- ποσά  
- ΑΦΜ δικαιούχων  
- υπογράφοντες  
- κωδικοί ΚΑΕ / ΑΛΕ  
- ημερομηνίες

Τα στοιχεία αποθηκεύονται σε ξεχωριστές στήλες ανά τύπο απόφασης
(`spending_*`, `commitment_*`, `direct_*`, `payment_*`).

Η συλλογή από τη Διαύγεια εκτελείται αυτόματα σε καθημερινή βάση μέσω GitHub Actions.

## Πηγή 3: Copernicus EFFIS

Τα δεδομένα για δασικές πυρκαγιές προέρχονται από το EFFIS API της υπηρεσίας  
Copernicus Emergency Management Service:

`api.effis.emergency.copernicus.eu`

Κάθε εγγραφή αντιστοιχεί σε ένα συμβάν πυρκαγιάς και περιλαμβάνει:

- `firedate` — ημερομηνία πυρκαγιάς  
- `area_ha` — καμένη έκταση σε εκτάρια  
- `shape` ή `centroid` — γεωμετρία συμβάντος  
- `lastupdate` — τελευταία ενημέρωση εγγραφής  
- στοιχεία κάλυψης γης  
- ποσοστό εντός Natura 2000

### Incremental fetch

Η συλλογή είναι incremental.

Το script αποθηκεύει το μέγιστο `lastupdate` που υπάρχει ήδη στη βάση και το χρησιμοποιεί ως σημείο εκκίνησης για το επόμενο run, με μικρό lookback window ώστε να πιάνονται διορθώσεις που έγιναν αναδρομικά.

Αν η βάση είναι κενή, γίνεται πλήρης φόρτωση από την αρχική ημερομηνία μέχρι σήμερα.

### Αντιστοίχιση σε δήμους

Κάθε πυρκαγιά αντιστοιχίζεται σε δήμο μέσω spatial join με τα όρια δήμων (GeoJSON).

Η διαδικασία είναι:

1. Αν υπάρχει πολύγωνο, χρησιμοποιείται το shape  
2. Αν δεν υπάρχει, χρησιμοποιείται το centroid  
3. Γίνεται reprojection σε EPSG:2100  
4. Υπολογίζεται επικάλυψη με δήμους

Αποθηκεύονται:

- municipality_key  
- overlap_ratio  
- match_method

## Αρχές επεξεργασίας δεδομένων

### Το raw dataset δεν αλλάζει

Τα αρχεία raw διατηρούνται όπως ήρθαν από την πηγή.  
Οι διορθώσεις γίνονται μόνο σε επόμενα στάδια, ώστε να υπάρχει audit trail.

Κάθε απόφαση καθαρισμού καταγράφεται σε αρχείο αποφάσεων.

### Φίλτρο σχετικότητας

Για τη Διαύγεια εφαρμόζεται φίλτρο που ελέγχει:

- λέξεις-κλειδιά στο subject  
- λέξεις-κλειδιά στο PDF

Αν ένα από τα δύο είναι θετικό, η εγγραφή κρατιέται.

### Ταξινόμηση φορέων

Οι φορείς ταξινομούνται με prefix matching σε κατηγορίες.

Υπάρχει επίσης blacklist για φορείς εκτός scope.

### Fallback ποσών

Αν λείπει ανάλυση ποσών, χρησιμοποιείται το συνολικό ποσό της απόφασης.

### Ασφάλεια τύπων

Πριν το enrichment, τα πεδία γίνονται cast σε object για να αποφευχθούν σφάλματα pandas.

## Πιθανά προβλήματα

- Λάθη x100 στα ποσά  
- Αποτυχία ανάγνωσης PDF  
- Πολύ γενικοί CPV  
- Ασυνέπεια ονομάτων φορέων  
- Centroid αντί για polygon σε μεγάλες πυρκαγιές

Δεν εφαρμόζονται αυτόματες διορθώσεις στο raw dataset.

## Πιθανά προβλήματα

### Λάθη x100 στα ποσά

Σε ορισμένες αποφάσεις τα ποσά φαίνεται να έχουν καταχωρηθεί χωρίς δεκαδικά, για παράδειγμα  
875.524,00 αντί για 8.755,24.  

Για τον εντοπισμό τέτοιων περιπτώσεων υπάρχει validator (`src/validate_diavgeia_amounts.py`) που συγκρίνει το ποσό της εγγραφής με το κείμενο του PDF.  
Δεν εφαρμόζονται αυτόματες διορθώσεις στο raw dataset. Το αρχικό αρχείο παραμένει όπως ήρθε από τη Διαύγεια.

Σε επόμενο στάδιο προβλέπεται χρήση manual override registry (π.χ. CSV ή YAML με αντιστοίχιση `ADA → corrected amount`), το οποίο θα εφαρμόζεται μόνο στο ingest layer.

### Αποτυχία ανάγνωσης PDF

Όταν ένα PDF δεν κατεβαίνει ή δεν μπορεί να αναλυθεί, το `pdf_match` είναι υποχρεωτικά False.  
Σε αυτές τις περιπτώσεις χρησιμοποιείται μόνο ο έλεγχος του subject ως fallback.

### Secondary CPVs και υπερίληψη

Ορισμένοι CPV, όπως `90910000-9`, είναι πολύ γενικοί.  
Το φίλτρο με λέξεις-κλειδιά στον τίτλο μειώνει τα άσχετα αποτελέσματα, αλλά δεν τα εξαλείφει πλήρως.

### Διαφορετική ονοματολογία φορέων

Δεν υπάρχει ενιαία γραφή ονομάτων φορέων.  
Όσοι δεν αναγνωρίζονται από τον classifier καταλήγουν στην κατηγορία  
`ΑΛΛΟΣ ΦΟΡΕΑΣ` και χρειάζονται χειροκίνητο έλεγχο.

### Copernicus centroid fallback

Σε εγγραφές χωρίς πλήρες πολύγωνο χρησιμοποιείται μόνο το centroid.  
Αυτό μπορεί να δώσει λάθος αντιστοίχιση σε μεγάλες πυρκαγιές που εκτείνονται σε περισσότερους από έναν δήμους.

## Testing

Η codebase περιλαμβάνει unit tests για τις βασικές λειτουργίες επεξεργασίας δεδομένων.  
Τα tests εκτελούνται με pytest (`testpaths = tests`) και αφορούν μικρές, απομονωμένες συναρτήσεις, όχι πλήρη end-to-end runs.

### Τι καλύπτουν τα tests

#### ΚΗΜΔΗΣ (`test_fetch_kimdis_procurements.py`)

Ελέγχεται ότι:

- το normalization αφαιρεί σωστά τόνους (`Δήμος Αθηναίων → δημος αθηναιων`)
- το business key προτιμά `referenceNumber` αντί για `diavgeiaADA`
- η dedupe λογική κρατά το νεότερο row ανά business key

Αυτό είναι κρίσιμο ώστε incremental runs να μην δημιουργούν διπλότυπα.

#### Relevance filter (`test_filter_relevance.py`)

Υπάρχουν τρία επίπεδα tests:

- unit test για το PDF lookup (κρατά το μεγαλύτερο κείμενο για duplicate ADA)
- integration test για `apply_relevance_filter`
- end-to-end test για `run_relevance_filter`

Το τελευταίο ελέγχει ότι γράφεται σωστά:

- το filtered CSV
- το raw CSV με νέες στήλες
- το run log

Χρησιμοποιείται `tmp_path` ώστε να μην δημιουργούνται αρχεία στο repo.

#### Copernicus (`test_fetch_copernicus.py`)

Ελέγχεται:

- bootstrap mode όταν η βάση είναι κενή
- incremental mode με lookback window
- φίλτρο ημερομηνιών για invalid `firedate`
- parsing helpers (bbox, JSON, booleans)

#### Municipality mapping (`test_map_copernicus_to_municipalities.py`)

Ελέγχεται η σωστή ανάλυση του `DATABASE_URL`, ειδικά σε περιπτώσεις όπου το env variable περιέχει ήδη το key.

#### Locate work (`test_locate_work.py`)

Ελέγχεται η deduplication λογική πριν το geocoding:

- συγχώνευση case-insensitive values
- συγχώνευση πολλών findings για το ίδιο σημείο
- επιλογή canonical ονόματος

### Φιλοσοφία testing

Τα tests ελέγχουν συμπεριφορά, όχι implementation.  
Χρησιμοποιείται `monkeypatch` για απομόνωση εξωτερικών εξαρτήσεων (DB, API).  
Τα fixtures στο `tests/fixtures/` είναι ελεγχόμενα δείγματα δεδομένων.

## Performance και σχεδιαστικές επιλογές

### Η βαριά επεξεργασία γίνεται στον server

Το frontend δεν κάνει aggregate queries.  
Όλα τα summaries υπολογίζονται στον server σε Postgres RPC functions και επιστρέφονται έτοιμα.

Η αρχική σελίδα φορτώνει με δύο βασικά calls:

- `get_hero_section_data(p_year_main, p_year_start)`
- queries για δήμους που εκτελούνται μόνο όταν ζητηθούν

Δεν μεταφέρεται raw dataset στον browser.

### Deduplication στον server

Σε incremental ingestion μπορεί να υπάρχουν duplicate rows.  
Η αποφυγή γίνεται στο SQL με:

```sql
ROW_NUMBER() OVER (PARTITION BY business_key ORDER BY id DESC)
```

Τα aggregates υπολογίζονται μόνο σε rows με `rn = 1`.

### Indexes

Υπάρχουν indexes σε:

- contract_signed_date
- firedate
- municipality_key
- lastupdate

ώστε να είναι γρήγορες οι χρονολογικές και γεωγραφικές αναζητήσεις.

### Dynamic year system

Τα έτη δεν είναι hardcoded.  
Ξεκινούν από `YEAR_START = 2024` και επεκτείνονται αυτόματα κάθε χρόνο.

### Fair YTD σύγκριση

Οι συγκρίσεις μεταξύ ετών γίνονται στο ίδιο χρονικό παράθυρο YTD.  
Το τέλος του παραθύρου υπολογίζεται με:

LEAST(τρέχουσα ημέρα, τελευταία ημέρα μήνα)

ώστε να μην επηρεάζουν δίσεκτα έτη.

### Ονομασία δικαιούχων

Το όνομα δικαιούχου επιλέγεται με χρονολογική προτεραιότητα:

- ΚΗΜΔΗΣ contractSignedDate
- Διαύγεια publishTimestamp
- fallback στον πίνακα beneficiary

Έτσι εμφανίζεται πάντα η πιο πρόσφατη επωνυμία.

## Pipeline επεξεργασίας

Το pipeline ακολουθεί διαδοχικά layers:

1. Raw collection  
   `fetch_kimdis_procurements.py`  
   `fetch_diavgeia.py`  
   `fetch_copernicus.py`

2. PDF pipeline  
   κατέβασμα και parsing → lookup `ADA → text`

3. Relevance filter  
   παράγει `2026_diavgeia_filtered.csv`

4. Copernicus spatial join  
   `map_copernicus_to_municipalities.py`

5. DB ingest  
   Supabase / Postgres

6. Frontend  
   React + Supabase + RPC

Τα ποσά στο frontend υπολογίζονται μετά από server-side deduplication, ώστε να αποφεύγεται double counting.
Για KIMDIS contract chains, το dedupe δεν βασίζεται μόνο σε business-key row ranking αλλά και στα `prevReferenceNo` / `nextRefNo`:
- superseded συμβάσεις μηδενίζονται στο `payment.amount_without_vat`
- superseded συμβάσεις εξαιρούνται και από `contract_count` / frontend contract lists
